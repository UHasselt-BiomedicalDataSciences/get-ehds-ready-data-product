The challenge

Legal permission ≠ usable data

The EHDS Regulation (EU) 2025/327 [1] obliges data holders across the EU to make electronic health data available for secondary use, and create the infrastructure for that data to be requested and used across borders. The regulation entered into force on 26 March 2025, and its obligations will become applicable from 26 March 2029. The window for data holders to get EHDS ready is now. But a legal right to access data is not the same as ready-to-use data.

The reality today

✕

Clinical data were collected to support care delivery, not designed for reuse in research.

✕

Different healthcare providers use their own data structures, local codes, and terminology, making direct comparison impossible.

✕

Often, there is no clearly defined owner for the data’s quality or lifecycle, and it ends up ageing in place, undocumented and unstable.

✕

Governance is bolted on defensively when sharing is requested, not designed in from the start.

What secondary use actually needs

✓

Data curated for specific research purposes, with clearly defined variables and agreed-upon meaning.

✓

A shared structure and vocabulary so data from different sources can be compared and combined.

✓

Clear ownership, version control, and a reliable interface that researchers can use to build pipelines.

✓

Quality and governance embedded in the data itself, not added as an afterthought.

The EHDS explicitly recognises secondary use as a core goal, aiming to enable health data to flow across organisational and national borders to support research, public health, policymaking, and innovation. The regulation defines what must be made available and discoverable, who may access which data categories, for which purposes, and under which conditions, and it even mandates dataset descriptions and a data quality and utility label. What it does not prescribe is how: the local data engineering needed to turn raw datasets into health data that can live up to those descriptions and labels.

The GET REHDY Technical Work Package targets precisely this gap between data collection and real‑world usefulness. What secondary use actually needs is data that behave like data products: discoverable, interoperable, and trustworthy assets that are purposefully designed for reuse in research, public health, and policymaking. By defining such health data products, together with the organisational and technical capabilities needed to govern, operate, and evolve them, GET REHDY Technical Work Package delivers the practical foundations that the EHDS legal framework requires but does not prescribe.

The paradigm shift

From data as a by-product to Data as a Product

The central concept in this work package is to treat health data not as something collected as a side effect of care, but as a product, intentionally designed, owned, and maintained to serve its users’ needs reliably.

01 / Curation

Designed for a purpose

Instead of collecting raw records in isolated silos, a data product is curated around specific user needs and use cases, defined before any code is written.

02 / Governance

Quality baked in

Governance is embedded into the data product itself, the quality rules, semantics, and access terms are part of the design, not added when someone asks to share it. This is the posture the EHDS data quality and utility label is designed to reward.

03 / Lifecycle

Clear ownership

Every data product has a named owner, a domain team accountable for its quality, evolution, and maintenance throughout its lifetime.

04 / Interface

A stable data contract

Consumers get a versioned, dependable interface they can build on. Internal systems can change freely, the published interface evolves only through explicit, versioned releases.

05 / Metadata

Catalogued & discoverable

A HealthDCAT-AP record, a dataset description conforming to the draft DCAT-AP extension for health data that underpins EHDS dataset descriptions [2], is prepared for publication in EHDS catalogues, giving consumers a stable, trustworthy description.

A well-designed data product makes data findable, accessible, interoperable, and reusable [3] and trustworthy (FAIR-T) in practice and is designed to lower the compliance burden by embedding good data management and governance into the data itself, rather than bolting it on when sharing is requested.

Step 1 — Design

The Data Product Canvas

A data product is not discovered in the data, it is intentionally designed. The Data Product Canvas [4] is a structured tool that defines all essential decisions on a single page, before a single line of code is written. It ensures that users, domain experts, and engineers share a common vision.

The canvas below shows the completed blueprint for our MS use case.

MS Data Product

Identifier: dp_ms_data

Domain

The neurology research team, which is the data product’s owner and is accountable for its quality, evolution, and maintenance.

Consumers & use cases

A researcher wants access to data for comparable analyses across studies and sites. At the same time, the OMOP CDM team behind the SwitchBox application needs the same data exposed through a stable, versioned interface so they can reliably transform it into OMOP CDM.

Data contract (output ports)

Versioned tables exposing standardised variables. Data holders can refactor internals freely; the interface changes only through explicit, versioned releases.

Sources (input ports)

Routinely collected clinical MS data, ingested from the institutional data warehouse (single input port).

Architecture

Input port (ingestion) → transformation (dbt) → Output ports

Ubiquitous language

Multiple Sclerosis Data Alliance (MSDA) Core Dataset (v2022) [5]: 44 globally recommended variables, each defined with SNOMED CT codes in a published data dictionary.

Classification

Source-aligned

The product exposes the neurology domain’s own clinical data in its ubiquitous language. It aggregates no other domains and is not tailored to a single consumer.

The canvas promotes a shift toward a data-product mindset by ensuring that each data product is valuable (creates research or economic value), usable (solves a genuine problem for the consumer) [6].

Step 2 — Build

Data contract & data contract-first design

A data contract is an agreement between a data producer and its consumers [7]. It specifies exactly what the data product exposes, its structure, semantics, quality rules, and service-level commitments, and is machine-readable, so it can be automatically enforced rather than just documented and forgotten.

Without a data contract

✕

An upstream change silently breaks a downstream dashboard or research pipeline and nobody is notified until the data user spots wrong numbers.

✕

Quality is checked manually, if at all, based on implicit and undocumented expectations.

✕

Discovering what a dataset contains means asking around, metadata lives in someone’s head or an outdated wiki page.

With a data contract

✓

Schema drift is detected automatically and blocks the change in the pipeline before it ever reaches a consumer.

✓

Quality rules and service-level agreements are validated automatically on every run, they pass or fail, with no ambiguity.

✓

Rich, machine-readable metadata fuels self-service discovery for both human and automated tools.

For the MS data product, this lifecycle is implemented with a concrete four-step engineering workflow:

01

Write

Author the ODCS contract defining the output interface

02

Build

Contract-first design: the dbt pipeline is built against the contract.

03

Enforce

At build time, the pipeline checks output against YAML definitions

04

Test

Quality rules from the contract are applied to the materialised data

Why machine-readable matters

Written in YAML (using the Open Data Contract Standard, ODCS) [8], the contract is version-controlled, validated automatically, and used to generate pipeline definitions.

What a data contract commits to

A data contract commits producers to deliver a well‑defined dataset interface (its schema and semantics, data quality, freshness and availability, ownership, and terms of use) that consumers can safely depend on, while allowing producers to change internal storage and processing as long as those commitments are upheld.

Step 3 — Scale to cross-border research

OMOP CDM & SwitchBox

A well-designed local data product is a necessary foundation, but it does not always enable cross-border research. For that, we need a shared data model so a researcher’s analytical code runs identically everywhere.

Harmonisation

OMOP CDM brings disparate sources (EHRs, claims, registries) into a single person-centric structure using shared concepts [9]. Results from different data holders can be pooled and compared.

Statistical power

Comparable data enables evidence generation across massive, combined cross-border populations, multiplying the research value of data that would otherwise remain siloed.

Data stays local

Standardised analyses are shipped to the data, not the other way around. Patient-level records never need to be centralised, it is the query that travels; not the data. This federated approach is proven at the European scale by the EHDEN network [10].

The SwitchBox

The SwitchBox [11], described in a preprint currently under peer review, is a pilot ETL (Extract, Transform, Load) tool that automates the conversion of the MS data product’s contracted output into the OMOP CDM format. It is the bridge between the local data product and cross-border research infrastructure.

Putting it together

The end-to-end workflow

01 / Design

Data Product Canvas

Data Product Canvas establishes purpose, ownership, consumers, and governance before engineering begins. Aligns all stakeholders on a shared blueprint.

02 / Build

Data contract-first design

The data contract enforces the output interface. The contract is the specification; the pipeline is built against it and tested on every run.

03 / Scale

SwitchBox & OMOP CDM

The SwitchBox ETL converts the data from the local data product into OMOP CDM format. The EHDS does not prescribe a common data model for secondary use; OMOP CDM is the model GET REHDY has chosen to close that syntactic and semantic gap [12], in line with large-scale European initiatives such as EHDEN [10] and the DARWIN EU network, which already standardise on OMOP.

04 / Publish

Findable

A HealthDCAT-AP [2] description of the dataset is prepared for publication in national and EU-level catalogues, enabling researchers to discover it and evaluate its suitability for their needs.
